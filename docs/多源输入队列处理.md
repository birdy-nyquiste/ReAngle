## 多源输入队列处理

本文档说明 ReAngle 改写功能中「多源输入队列」的整体设计：用户可以将来自不同来源的内容（纯文本、文件、URL、YouTube 等）依次加入队列，系统会在后端统一抽取与清洗，再一次性发送给所选模型进行改写。

---

## 一、用户视角的使用方式

在主应用页面（`MainApp`）左侧，用户可以通过四种方式添加内容：

1. **Text**：直接粘贴或输入纯文本片段。  
2. **File**：上传 `.docx`、`.pdf`等文件。  
3. **URL**：粘贴网页链接，由系统抓取正文。  
4. **YouTube**：粘贴视频链接，当前按链接文本纳入上下文。  

每次在对应 Tab 中填写好内容后点击「Add to Queue」，该输入会作为一条独立记录加入队列列表（右侧显示简要信息，例如标题/文件名、字符数等）。用户可以多次添加、多种类型混合添加，也可以从队列中删除不需要的条目。  

当队列准备完毕、配置好改写模型与风格后，点击「Transform Content」，前端会把整个队列序列化后提交给后端，由后端负责统一抽取、聚合和改写，最终得到一份综合后的改写结果与摘要。

---

## 二、前端实现概要

前端的输入队列逻辑主要集中在 `frontend/src/pages/MainApp.tsx`：

- 使用 `inputItems: InputItem[]` 维护当前队列，每个元素包含 `id`、`type`（`text` / `file` / `url` / `youtube`）、`content` 以及 `meta`（标题、摘要信息）。
- 不同 Tab 对应不同输入方式：
  - 文本与 URL 直接将字符串写入 `content`；
  - 文件类型会在前端只保存 `File` 对象引用，真正上传在提交时处理。
- 点击「Add to Queue」时，根据当前 Tab 和输入内容构造 `InputItem` 并追加到队列，同时清空该输入框。
- 在点击「Transform Content」时：
  - 创建 `FormData` 并遍历 `inputItems`：
    - 对于文件：将 `File` 附加到 `FormData`，记录 `contentKey`；
    - 对于其他类型：直接将文本内容放到序列化对象中。
  - 将上述对象数组序列化为 JSON 字符串，放入表单字段 `inputs` 中一并提交给后端。

这样设计的好处是：前端只需维护一个统一的队列结构，不同来源的输入在「添加」和「提交」两个阶段被规范化，后端拿到 `inputs` 后可以用同一套逻辑进行拆解和抽取。

---

## 三、后端抽取与聚合流程

后端的主要逻辑位于 `app/routers/rewrite.py` 的 `rewrite_article` 路由：

1. **解析队列**：
   - 从表单字段 `inputs` 中读取 JSON 字符串并解析为 `List[Dict[str, Any]]`。
   - 若解析失败（JSON 结构错误或字段缺失），抛出 `ContentExtractionError("Invalid inputs format")`，并在日志中记录 `request_id` 与原始异常。

2. **按类型抽取内容**：遍历 `items`，根据 `type` 进行分派：  
   - `text`：直接读取 `content`，加上简单标记（如 `[文本]`）。  
   - `url`：调用 `extract_text_from_url` 抓取正文，失败时记录警告并回退为原始 URL 文本。  
   - `youtube`：当前以原始链接字符串加入上下文（后续可扩展为字幕抓取）。  
   - `file`：
     - 通过 `contentKey` 在表单中找到对应上传文件；
     - 根据文件后缀调用不同的抽取器：
       - `.docx` → `extract_text_from_docx`
       - `.pdf` → `extract_text_from_pdf`
       - 图片（`.png` / `.jpg` / `.jpeg` / `.webp`）→ `extract_text_from_image`
       - 其它：按文本尝试解码（UTF-8 / GBK），失败时进行容错处理；
     - 抽取成功的内容会带上文件名标记（如 `[文件] xxx.pdf`）。

3. **内容聚合**：
   - 将各条抽取结果按固定分隔符（例如 `\n\n---\n\n`）拼接为一段长文本 `clean_text`，用于后续改写。
   - 若最终 `clean_text` 为空（所有输入都抽取失败或为空），抛出 `ContentExtractionError("Extracted text is empty or invalid")`。

4. **调用改写模型**：
   - 将 `clean_text` 和用户在前端设置的指令 `prompt` 交给 `rewriting_client`，根据选定的 LLM 类型（OpenAI / Gemini / Qwen 等）进行摘要和改写。
   - 返回的结果包含三部分：原始聚合文本 `original`、摘要 `summary` 与改写后的正文 `rewritten`，统一封装为 `RewriteResponse` 返回给前端。

## 四、设计动机与优势

- **统一改写上下文**：用户不需要关心不同来源的内容如何合并，只需把相关材料逐条加入队列，系统会在后端按顺序聚合为一个统一上下文，让模型能够在同一轮中理解和改写。
- **扩展性好**：新增一种输入来源时（例如小程序、微信公众号文章等），只要在队列结构中增加新的 `type`，在后端抽取阶段实现对应的抽取器，即可无缝接入，不影响现有改写流程。
- **可观测性与容错**：抽取流程中对每一步都记录日志；单个 URL/文件失败不会直接导致整次请求失败（视场景决定是跳过还是抛错），并通过自定义异常与统一错误响应体系，给前端提供稳定的反馈。

通过多源输入队列处理机制，ReAngle 能够自然支持「多篇文章合并改写」「文档 + 网页 + 自己笔记组合」等实际使用场景，为后续更多内容形态的接入（如小程序、视频字幕等）提供了清晰的扩展路径。
