# 账号登录与订阅计费功能说明

本文档从产品与工程角度说明 ReAngle 的账号体系、登录与路由保护、用量与 Free/Pro 概念、Stripe 订阅与支付流程及用户可见的用量展示；技术实现与表结构见《用户系统技术文档》。

---

## 一、账号与登录流程

ReAngle 采用 **邮箱 + 密码** 的账号体系，由 Supabase 提供认证服务：

- **注册**：用户在注册页填写邮箱和密码，前端调用 Supabase 的 `signUp` 接口；注册成功后，Supabase 向邮箱发送验证链接，用户点击后账号才正式激活。
- **登录**：在登录页输入邮箱和密码，前端调用 `signInWithPassword`，成功后获得一份 `session`（包含 `access_token`），存入前端 `AuthContext`；后续所有需要鉴权的 API 请求都会在 HTTP Header 中携带 `Authorization: Bearer <access_token>`。
- **会话保持与退出**：
  - 前端在应用初始化时通过 `supabase.auth.getSession()` 检查本地是否已有有效会话，并通过 `onAuthStateChange` 监听登录/登出事件；
  - 用户点击「Sign Out」时，调用 `supabase.auth.signOut()` 并清空本地状态。
- **路由保护**：`/app`（主应用）、`/profile`（个人中心）等页面使用 `ProtectedRoute` 组件，未登录用户访问时会被重定向到 `/login`，避免在未认证状态下调用改写或查看用量。

在后端，所有需要登录的路由（例如 `/api/v2/inputs/`、`/api/v2/reangle/`、支付与用量查询等）都通过依赖 `get_current_user` 对 `access_token` 做 Supabase 服务端校验，确认未过期、未被撤销后再执行业务逻辑。

## 二、用量与订阅的基本概念

ReAngle 将用户分为两类：

- **免费用户（Free）**：每个月拥有固定次数的改写额度（例如 5 次），达到上限后无法再调用改写接口。
- **订阅用户（Pro）**：通过 Stripe 订阅 Pro 计划后，获得更高或无限制的用量（例如 `usage_limit = -1` 表示无限）。

每个用户在 Supabase `profiles` 表中有一条记录，核心字段包括：

- `usage_count`：当前计费周期内已经使用的改写次数；
- `usage_limit`：当前计划允许的最大次数（-1 表示无限制）。

在需要消耗额度的路由（例如 `/api/v2/inputs/`、ReAngle 相关调用）上，后端通过依赖 `check_usage_limit` 做用量控制：

1. 从请求 Header 中解析 `access_token`，通过 `get_current_user` 验证并拿到 `user_id`；
2. 调用 Supabase RPC 函数 `increment_usage(user_id)`，在数据库中原子性递增 `usage_count` 并判断是否超过 `usage_limit`；
3. 若返回 `false`，则直接返回 HTTP 402（Payment Required），前端据此向用户提示「当月用量已用完，前往升级或下月再试」；若返回 `true`，则放行改写逻辑。

用量重置由 Stripe 订阅周期驱动：当 Stripe 产生续费成功的 `invoice.payment_succeeded` 事件时，后端会通过 Webhook 把对应用户的 `usage_count` 重置为 0，并更新 `current_period_end` 等字段。

## 三、订阅计划与支付流程

订阅计费基于 **Stripe Checkout + Customer Portal** 实现，核心目标是尽量少写自定义支付逻辑，将状态管理交给 Stripe 与 Supabase：

1. **发起订阅**：
   - 用户在前端的 Pricing 页面点击「Upgrade to Pro」按钮；
   - 前端携带当前用户的访问 Token 调用 `/api/v2/payment/create-checkout-session`；
   - 后端根据用户是否已有有效订阅决定：
     - 若已有 `active` / `trialing` 订阅，则直接创建 Customer Portal 会话，引导用户管理现有订阅；
     - 若没有订阅，则为该用户创建新的 Stripe Checkout Session，并返回重定向 URL。

2. **支付完成与订阅同步**：
   - 用户在 Stripe 页面完成支付或修改订阅方案后，Stripe 会向后端的 `/api/v2/payment/webhook` 发送事件（例如 `checkout.session.completed`、`customer.subscription.updated`）；
   - 后端使用「State-Based Sync」模式处理 Webhook：
     - 仅将事件视作「有变化」的通知，而不是直接信任事件负载中的所有字段；
     - 收到通知后，通过 Stripe API 查询最新的 Subscription 状态，并同步写入 Supabase `subscriptions` 与 `profiles.usage_limit`。

3. **订阅状态与用量展示**：
   - 前端 Profile 页通过 `/api/v2/payment/usage` 获取当前用户的用量和订阅信息（是否为 Pro、当前周期起止时间、剩余额度等），并展示在「Plan & Usage」区域；
   - 若用户没有订阅，则展示 Free 计划信息与引导按钮跳转定价页；若已订阅，则提供「Manage Subscription」按钮打开 Stripe Customer Portal，方便用户自行升级/降级/取消。

整个支付链路的安全性依赖于：前端只持有 Supabase `anon` 公钥与 Stripe 公钥，所有敏感操作（校验 JWT、调用 Stripe Secret Key、处理 Webhook）都仅在后端执行；同时，通过用量计数与订阅状态的结合，确保只有登录且未超限的用户才能继续调用改写与相关增值功能（如 Avatar 数字人播报）。
