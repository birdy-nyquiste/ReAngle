# 多模型调用配置与路由说明

本文档说明 ReAngle 如何通过**用户设置**中的模型与系统 Prompt、以及后端 ReAngle 服务，支持多模型改写（如 ChatGPT、Gemini、Qwen）；当前改写入口为 **`POST /api/v2/reangle/`**，模型由设置中的 `reangle_model` 决定。

ReAngle 的改写能力支持多种大模型（如 OpenAI、Gemini、Qwen）。用户在**设置页**选择 ReAngle 使用的模型与系统 Prompt并保存；执行 ReAngle 时后端从用户设置读取模型，将请求路由到对应提供商，返回统一的 `summary` 与 `rewritten_content`。

---

## 一、整体调用链路概览

> 用户先在设置页选择并保存 ReAngle 模型（及可选系统 Prompt）→ 完成 Inputs（多源输入入队）与可选 DeAngle → 前端提交 **`POST /api/v2/reangle/`**（携带 `prompt`、`selected_facts`、`selected_angles`）→ 后端从 Session 取 `clean_text` 与 DeAngle 结果，从用户设置取 `reangle_model` 与 `reangle_system_prompt`，调用 ReAngle 服务 → 服务层按模型路由到对应 LLM 客户端，返回 `summary` 与 `rewritten_content`。

模型标识不随单次请求传递，而是**存储在用户设置中**，由后端在 ReAngle 时读取。

---

## 二、前端模型选择与设置

用户在**设置页**（如 `SettingsPage`）中配置：

- **ReAngle 模型**：下拉选择使用的模型（如对应 OpenAI / Gemini / Qwen 的模型 ID）。
- **系统 Prompt**（可选）：ReAngle 使用的系统级提示词。

设置通过 **`GET/PUT /api/v2/settings/`** 读写；提交 ReAngle 时前端只传 `prompt`、`selected_facts`、`selected_angles`，不传 `llm_type`。

---

## 三、后端 ReAngle 路由与请求整理

后端改写入口为 **`app/routers/v2/reangle.py`** 的 **`POST /api/v2/reangle/`**：

- 从 Session 获取 `clean_text`（由 Inputs 步骤写入）及 DeAngle 结果（`facts`、`angles`）；无则报错提示先完成 Inputs/DeAngle。
- 从 **用户设置** 获取当前用户的 `reangle_model`、`reangle_system_prompt`（如 `app/services/settings` 的 `get_settings(user_id)`）。
- 调用 **`app/services/re/reangle_service.run_reangle`**，传入 `clean_text`、用户选中的 `selected_facts`/`selected_angles`、`prompt`、`model`、`system_prompt`。
- 返回 **`ReAngleResponse`**：`summary`、`rewritten_content`。

路由不直接调用各 LLM SDK，由 ReAngle 服务内部按 `model` 路由到对应实现。

---

## 四、服务层多模型路由（ReAngle 服务）

ReAngle 服务（如 `reangle_service`）根据传入的 `model`（来自用户设置）选择对应 LLM 客户端，调用其生成接口，得到统一的 `summary` 与 `rewritten_content`。若 `model` 未知则抛出 `InvalidInputError`。各 LLM 客户端实现相同风格的接口（如接收原文、用户 prompt、系统 prompt，返回摘要与正文），上层不感知具体提供商差异。

---

## 五、各模型的配置与调用要点

### 5.1 ChatGPT 5（OpenAI）

- **配置**：
  - 环境变量：`OPENAI_API_KEY`（必填，否则无法使用该模型）。
  - 可选地配置自定义 API Base 或代理（若使用兼容 OpenAI 协议的网关）。
- **调用方式**：
  - 使用官方/兼容的 Chat Completions 接口，模型 ID 对应前端的 `gpt-5-mini` 或其它映射。
  - Prompt 结构通常为：
    - system：约束角色与输出格式（摘要 + 改写规则）；
    - user：包含指令 `instruction` 与原文 `source`。
- **错误处理**：网络错误、认证失败、配额超限会统一包装为 `LLMProviderError` 并被全局异常处理器捕获。

### 5.2 Gemini

- **配置**：
  - 环境变量：`GEMINI_API_KEY`。
- **调用方式**：
  - 使用官方 Gemini SDK 或 HTTP API，模型 ID 映射为前端的 `gemini-3-flash-preview` 等。
  - 将 `instruction` + `source` 组织为一个或多个 prompt part，遵循 Gemini 的消息格式。
- **错误处理**：类似 OpenAI，所有上游异常被 `LLMProviderError` 统一封装。

### 5.3 Qwen

- **配置**：
  - 环境变量：`DASHSCOPE_API_KEY` 或其他 Qwen 相关 Key（取决于接入方式）。
- **调用方式**：
  - 使用阿里 DashScope 或官方 Qwen SDK，模型 ID 与前端的 `qwen-flash` 对应。
  - 将 `instruction` 和 `source` 拼接为 prompt，或通过多段 message 传入。
- **错误处理**：与其他模型一致，通过 `LLMProviderError` 统一对外暴露。

---

## 六、与其他功能的关系

目前多模型配置主要应用在「文章改写 + 摘要」这条主链路中：

- **数字人播报**：依赖改写结果和口播稿，底层数字人视频生成使用 HeyGen（见《数字人播报》），与选择哪家 LLM 改写文章相互独立。
- **口播稿生成（Voiceover）**：当前实现仅使用 OpenAI 一个提供商（统一的口播稿系统 Prompt），未做多模型切换；如需扩展，可复用本文件描述的 `llm_type` 路由模式。

这种设计保证：改写主流程可以灵活切换和扩展大模型，而其他上层功能（Avatar、TTS、小程序等）只需要消费统一的改写结果即可。

---

## 七、扩展新模型的步骤

若需接入新模型（如 Claude 或自建），建议：

1. 在环境变量中新增该模型的 API Key，并在启动或调用前校验。
2. 在 ReAngle 服务所用的 LLM 封装层中新增对应客户端，实现与现有模型一致的接口（如接收原文、prompt、system_prompt，返回 summary 与正文）。
3. 在用户设置的模型选项中增加新模型 ID（如写入配置或数据库的允许列表），并在 ReAngle 服务路由表中将该 ID 映射到新客户端。
4. 在前端设置页的模型下拉框中增加该选项。
5. 视需要补充该模型的错误码或限流说明。

这样多模型调用仍由「用户设置中的模型 ID + 后端统一路由」完成，便于后续扩展与切换默认模型。
