## 多模型调用配置与路由说明

ReAngle 的核心改写能力支持三种主流大模型：**ChatGPT 5（OpenAI）**、**Gemini（Google）**、**Qwen（阿里）**。本功能通过统一的数据结构与服务层封装，让前端可以在一个下拉框中自由切换模型，而后端则根据 `llm_type` 将请求路由到对应的提供商，并返回一致结构的结果。

---

## 一、整体调用链路概览

完整调用链可以概括为：

> 前端选择模型并提交改写请求 → 后端 `/rewrite` 路由接收并整理多源输入 → 将统一的 `instruction + source` 和 `llm_type` 交给 `rewriting_client` → `rewriting_client` 根据 `llm_type` 调用相应大模型客户端（OpenAI / Gemini / Qwen）→ 返回统一格式的 `summary + rewritten` 给前端展示。

整个过程中，**只有一处地方依赖前端传入的模型标识**（`llm_type`），后端其余模块都通过内部的映射表进行路由和封装。

---

## 二、前端模型选择与请求结构

在前端主应用页面 `MainApp.tsx` 中，用户在左侧「Configuration」区域可以看到一个 **AI Model** 下拉框：

- 选项示例（实际文案通过多语言文案控制）：
  - `gpt-5-mini`（ChatGPT 5 / OpenAI）
  - `gemini-3-flash-preview`（Gemini）
  - `qwen-flash`（Qwen）
- 组件层面通过一个 `model` 状态变量记录当前选择，并在提交改写请求时一并发送给后端。

当用户点击「Transform Content」时，前端会：

1. 把多源输入队列序列化为 JSON，写入 `FormData` 字段 `inputs`；
2. 将当前的改写指令写入 `prompt` 字段；
3. 将选择的模型标识（`model`）写入 `llm_type` 字段；
4. 调用 `POST /api/v1/rewrite`，将上述表单提交给后端。

前端并不关心各个大模型的 API 细节，只负责提供一个简单的「模型类型字符串」，其余逻辑全部在服务端完成。

---

## 三、后端路由与请求整理

后端改写入口为 `app/routers/rewrite.py` 中的：

```python
@rewrite_router.post("", response_model=RewriteResponse)
async def rewrite_article(request: Request, rewrite_request: Annotated[RewriteRequest, Form()], ...):
    ...
```

该路由主要做三件事：

1. **解析多源输入队列**：从 `rewrite_request.inputs` 中读取 JSON，结合上传文件表单，统一抽取出 `clean_text`（详见《多源输入队列处理》文档）。
2. **构造改写调用参数**：从 `rewrite_request` 中拿到：
   - `llm_type`：前端选择的模型类型字符串；
   - `prompt`：用户在前端填写的风格/指令；
   - `clean_text`：上一步抽取并聚合后的长文本。
3. **调用统一改写服务**：
   - 调用 `rewriting_client.get_rewriting_result(llm_type=..., instruction=prompt, source=clean_text)`；
   - 得到的结果统一包含 `summary` 和 `rewritten`；
   - 封装为 `RewriteResponse(original=clean_text, summary=..., rewritten=...)` 返回前端。

路由本身不直接与 OpenAI/Gemini/Qwen 的 SDK 交互，而是依赖 `rewriting_client` 这一中间层以保持解耦。

---

## 四、服务层多模型路由（rewriting_client）

`app/services/llms/rewriting_client.py`（具体文件名略）是多模型调用的核心，它负责：

1. 定义一个统一的接口 `get_rewriting_result(llm_type, instruction, source)`；
2. 维护一个从 `llm_type` 到具体客户端的路由表，例如：
   - `"gpt-5-mini"` → OpenAI 客户端  
   - `"gemini-3-flash-preview"` → Gemini 客户端  
   - `"qwen-flash"` → Qwen 客户端
3. 根据传入的 `llm_type` 选择对应实现，若传入未知类型，则抛出 `InvalidInputError("Unsupported llm_type: ...")`。

每个具体的大模型客户端都实现了类似的接口，例如：

```python
class OpenAIRewritingClient:
    async def rewrite(self, instruction: str, source: str) -> RewritingResult: ...

class GeminiRewritingClient:
    async def rewrite(self, instruction: str, source: str) -> RewritingResult: ...

class QwenRewritingClient:
    async def rewrite(self, instruction: str, source: str) -> RewritingResult: ...
```

`RewritingResult` 是一个统一的数据结构，至少包含：

- `summary: str`：对输入内容的简要摘要；
- `rewritten: str`：经过风格化与结构优化之后的正文。

`rewriting_client` 对外只暴露统一的 `get_rewriting_result`，上层路由不会感知具体提供商的差异。

## 五、各模型的配置与调用要点

### 5.1 ChatGPT 5（OpenAI）

- **配置**：
  - 环境变量：`OPENAI_API_KEY`（必填，否则无法使用该模型）。
  - 可选地配置自定义 API Base 或代理（若使用兼容 OpenAI 协议的网关）。
- **调用方式**：
  - 使用官方/兼容的 Chat Completions 接口，模型 ID 对应前端的 `gpt-5-mini` 或其它映射。
  - Prompt 结构通常为：
    - system：约束角色与输出格式（摘要 + 改写规则）；
    - user：包含指令 `instruction` 与原文 `source`。
- **错误处理**：网络错误、认证失败、配额超限会统一包装为 `LLMProviderError` 并被全局异常处理器捕获。

### 5.2 Gemini

- **配置**：
  - 环境变量：`GEMINI_API_KEY`。
- **调用方式**：
  - 使用官方 Gemini SDK 或 HTTP API，模型 ID 映射为前端的 `gemini-3-flash-preview` 等。
  - 将 `instruction` + `source` 组织为一个或多个 prompt part，遵循 Gemini 的消息格式。
- **错误处理**：类似 OpenAI，所有上游异常被 `LLMProviderError` 统一封装。

### 5.3 Qwen

- **配置**：
  - 环境变量：`DASHSCOPE_API_KEY` 或其他 Qwen 相关 Key（取决于接入方式）。
- **调用方式**：
  - 使用阿里 DashScope 或官方 Qwen SDK，模型 ID 与前端的 `qwen-flash` 对应。
  - 将 `instruction` 和 `source` 拼接为 prompt，或通过多段 message 传入。
- **错误处理**：与其他模型一致，通过 `LLMProviderError` 统一对外暴露。

## 六、与其他功能的关系

目前多模型配置主要应用在「文章改写 + 摘要」这条主链路中：

- **Avatar 数字人播报**：依赖改写结果和口播稿，底层数字人视频生成使用 HeyGen（见《Avatar数字人播报功能》），与选择哪家 LLM 改写文章相互独立。
- **口播稿生成（Voiceover）**：当前实现仅使用 OpenAI 一个提供商（统一的口播稿系统 Prompt），未做多模型切换；如需扩展，可复用本文件描述的 `llm_type` 路由模式。

这种设计保证：改写主流程可以灵活切换和扩展大模型，而其他上层功能（Avatar、TTS、小程序等）只需要消费统一的改写结果即可。

---

## 七、扩展新模型的步骤

如果后续需要接入新的模型（例如 Claude 或自建模型），推荐按照以下步骤：

1. 在环境变量中新增对应的 API Key 配置，并在启动时检查必填项是否存在。
2. 在 `app/services/llms` 下新增对应的客户端实现，提供 `rewrite(instruction, source) -> RewritingResult` 接口。
3. 在 `rewriting_client` 中更新路由表，将新的 `llm_type` 字符串映射到该客户端。
4. 在前端的模型下拉框中增加选项，并将其值设置为同一个 `llm_type` 字符串。
5. 视需要在错误处理文档中补充该模型的特定错误码或限流行为。

通过上述约定，多模型调用在代码中始终保持「前端只关心一个字符串，后端统一封装差异」的结构，方便后续持续演进与切换默认模型策略。
