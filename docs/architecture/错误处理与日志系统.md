# 错误处理与日志系统文档

本文档概述了 ReAngle 项目中实现的新错误处理和日志架构。该系统使用 `loguru` 进行结构化、上下文感知的日志记录，并使用集中的 FastAPI 异常处理器来提供一致的 API 错误响应。

## 1. 系统概述

- **统一日志**：使用 `loguru` 替代散落的 `print()` 语句，提供结构化日志（`INFO`, `ERROR`, `DEBUG`）。
- **集中错误处理**：所有异常都由 `app/core/handlers.py` 中的全局处理器捕获，并转换为标准的 JSON 格式。
- **请求追踪**：每个请求都会被分配一个唯一的 `request_id`，该 ID 会出现在所有相关的日志条目中。

## 2. 日志架构

### 配置

- **文件**: `app/configs/logger.py`
- **库**: `loguru`

### 特性

1. **拦截**: 标准 Python `logging` 消息（来自 Uvicorn, FastAPI）会被拦截并重定向到 Loguru，确保统一的日志格式。
2. **请求上下文**:
    - 中间件为每个请求生成一个 `request_id`。
    - `logger.configure(extra={"request_id": "N/A"})` 确保请求上下文之外的日志（如启动日志）具有默认 ID。
3. **输出**:
    - **控制台**: 适合开发的人类可读、彩色日志。
    - **文件**: 存储在 `logs/app.log` 中的结构化日志。
        - **轮转**: 每 10 MB。
        - **保留**: 10 天。

### 日志格式

```text
YYYY-MM-DD HH:mm:ss | LEVEL    | request_id | module:function:line - Message
```

## 3. 中间件

- **文件**: `app/middleware/request_logging.py`
- **类**: `RequestLoggingMiddleware`

该中间件执行三个关键功能：

1. **ID 生成**: 为每个传入的 HTTP 请求生成一个 UUID4 `request_id`。
2. **上下文绑定**: 将此 ID 绑定到 `loguru` 上下文，确保请求生命周期内生成的所有日志都包含此 ID。
3. **生命周期日志**:
    - 记录请求详情：`方法`, `URL`, `客户端 IP`。
    - 记录响应详情：`状态码`, `处理时间` (ms)。
    - 向响应添加 `X-Request-ID` 头。

## 4. 异常处理架构

所有自定义异常都继承自一个基类以确保一致性。

### 基类异常

- **文件**: `app/core/exceptions.py`
- **类**: `AppException`
- **属性**:
  - `message`: 人类可读的错误描述。
  - `status_code`: HTTP 状态码 (例如 400, 500)。
  - `code`: 内部错误代码字符串 (例如 `INTERNAL_ERROR`)。
  - `details`: 可选的额外上下文信息字典。

### 具体异常

| 异常 | HTTP 状态 | 用途 |
| :--- | :--- | :--- |
| `ContentExtractionError` | 422 (Unprocessable Entity) | 解析 URL、PDF 或 DOCX 文件失败。 |
| `LLMProviderError` | 502 (Bad Gateway) | OpenAI 或 Gemini API 调用失败（网络、认证等）。 |
| `InvalidInputError` | 400 (Bad Request) | 验证失败（例如缺少文件、文本为空）。 |

### 全局处理器

- **文件**: `app/core/handlers.py`
- 在 `app/main.py` 中注册。

1. **`app_exception_handler`**: 捕获 `AppException` 并返回标准 JSON 错误响应。
2. **`validation_exception_handler`**: 覆盖 FastAPI 默认的验证错误 (422) 以匹配我们的 JSON 模式。
3. **`global_exception_handler`**: 捕获任何未处理的 `Exception`，通过 `logger.exception` 记录完整堆栈跟踪，并返回通用的 500 错误。

## 5. 标准化 API 响应

所有错误都返回 `app/schemas/error_response_schema.py` 中定义的一致 JSON 结构。

### 模式

```python
class BaseErrorResponse(BaseModel):
    success: bool = False
    error: str
    code: Optional[str]
    details: Optional[Dict[str, Any]]
```

### JSON 示例

```json
{
    "success": false,
    "error": "Failed to download URL",
    "code": "CONTENT_EXTRACTION_ERROR",
    "details": {
        "url": "https://example.com"
    }
}
```

## 6. 服务层集成

### 提取器 (`app/services/extractors.py`)

- 使用 `logger.debug()` 进行流程追踪。
- 捕获 `httpx` 或库错误并抛出 `ContentExtractionError`。
- **不再返回错误字符串。**

### LLM 客户端 (`app/services/llms/*.py`)

- 使用 `logger.info()` 追踪 API 使用情况（模型、提供商）。
- 在调用前验证 API 密钥和配置。
- 捕获网络/API 错误并抛出 `LLMProviderError`。

## 7. 开发人员使用指南

1. **日志记录**:
    始终使用 `loguru`。不要使用 `print()`。

    ```python
    from loguru import logger

    logger.info("Processing started")
    logger.error("Something went wrong", some_var=123)
    ```

2. **抛出错误**:
    抛出 `app.core.exceptions` 中的具体异常。不要返回错误字典或字符串。

    ```python
    from app.core.exceptions import ContentExtractionError

    if not file.filename.endswith('.pdf'):
        raise ContentExtractionError("Only PDF files are supported")
    ```

3. **捕获错误**:
    让全局处理器处理响应格式化。只有当你能从异常中恢复，或者需要将其包装在更具体的自定义异常中时，才捕获异常。

## 8. 主要业务模块的错误处理约定

随着功能增多，每个业务模块都在「日志 + 自定义异常」的基础上增加了自己的约定，本节按功能做一个集中说明，便于排查问题与后续扩展。

### 8.1 改写主流程（`/rewrite`)

- **输入解析与内容抽取**：`rewrite_router` 在解析队列输入（文本、URL、文件、YouTube）时，如果 JSON 结构非法会抛出 `ContentExtractionError("Invalid inputs format")`，`details` 中包含 `request_id` 和原始异常信息；URL / 文件抽取失败会写入 `logger.warning` 或 `logger.error`，并按需抛出 `ContentExtractionError`，日志里会带上 URL 或文件名。
- **调用 LLM 改写**：`rewriting_client` 在调用上游 LLM 前会用 `logger.info` 记录 `provider`、`prompt_len`、`source_len` 等信息；调用失败（网络、认证、模型异常等）统一抛出 `LLMProviderError("Failed to generate rewrite: ...")`，并在 `details` 中补充 `llm_type`、`prompt_len`、`source_len` 与 `request_id`。

### 8.2 语音与口播稿（TTS / Voiceover）

- **TTS（`/rewrite/tts`）**：入口使用 `logger.info` 记录文本长度；调用 `tts_client` 失败时捕获异常并抛出 `LLMProviderError("TTS generation failed: ...")`，前端只展示友好文案，具体错误原因通过后端日志排查。
- **口播稿生成（`/rewrite/voiceover`）**：对输入文本 `strip()` 后为空时抛出 `InvalidInputError("Text is empty, cannot generate voiceover script.")`；`voiceover_client` 调用失败时抛出 `LLMProviderError("Voiceover generation failed: ...")`；若返回结果为空字符串则抛出 `LLMProviderError("Voiceover script is empty.")`，避免前端拿到不可用脚本。

### 8.3 数字人视频生成（Avatar / Video Agent）

- **AvatarClient（`/rewrite/avatar` → `avatar_client`）**  
  - 入口：缺少 `HEYGEN_API_KEY` 时抛出 `LLMProviderError("Missing HEYGEN_API_KEY environment variable")`；口播稿为空时抛出 `LLMProviderError("Empty text provided for avatar generation")`。
  - 生成阶段：调用 `/v2/video/generate`，非 200 响应会记录完整响应体并抛出 `LLMProviderError("HeyGen API generation error: ...")`；响应中缺少 `data.video_id` 会抛出 `LLMProviderError("Invalid response from HeyGen API")`。
  - 轮询阶段：定期调用 `/v1/video_status.get`，非 200 响应用 `logger.warning` 记录后重试；`status == "failed"` 时抛出 `LLMProviderError("Video processing failed: ...")`；长时间无结果则记录 `last_status` 并抛出 `LLMProviderError("Avatar video generation timed out...")`。
- **VideoAgentClient（`/rewrite/video-agent` → `video_agent_client`）**  
  - 在入口对文本长度做上限检查（例如 4000 字符），超出抛出 `InvalidInputError`，避免将超大 prompt 直接推给 HeyGen。
  - 调用 `/v1/video_agent/generate` 时，HTTP 非 200 或 `code != 100`、响应里包含 `error` 字段时，都会封装为 `LLMProviderError("HeyGen Video Agent: ...")`；轮询 `/v1/video_status.get` 的逻辑与 AvatarClient 类似，超时时会在错误信息中明确提示「Video Agent generation timed out，可能仍在后台处理」。

### 8.4 视频下载代理（`/rewrite/video-download`）

- 对 `url` 做严格校验：为空、非 `https`、或主机名不在允许的 HeyGen 域名白名单中时，分别抛出 `InvalidInputError("Missing url")`、`InvalidInputError("Only https url is allowed")`、`InvalidInputError("Unsupported download host")`，防止 SSRF 风险。
- 真正下载阶段：非 200 响应抛出 `LLMProviderError("Download failed: HTTP ...")`；超过最大文件大小（默认 300MB）时抛出 `LLMProviderError("Video file too large to proxy.")`；`httpx.TimeoutException` 或其他网络错误对应抛出 `LLMProviderError("Video download timed out...")` 与 `LLMProviderError("Download network error: ...")`。

## 9. 前后端协作约定

- 后端返回的 `error` 文案尽量保持简洁、面向用户，详细原因和第三方错误码通过日志记录；前端只根据 `success=false` 与 `error` 字段展示信息，不做复杂的 HTTP status 分支。
- 所有可预期的输入问题、配置问题、第三方错误，都优先用 `InvalidInputError` / `ContentExtractionError` / `LLMProviderError` 显式抛出，而不是依赖未捕获的通用异常落到 500。

---

## 10. 新功能接入错误处理的 Checklist

当新增一个功能（例如新的外部 API、支付方式或导出能力）时，建议按以下步骤接入现有错误与日志系统：

1. 在服务层创建独立模块（如 `app/services/xxx_client.py`），在关键路径前后加上 `logger.info` / `logger.error` / `logger.warning`。
2. 对用户输入做基本校验，出现明显错误时使用 `InvalidInputError` 返回 4xx，而不是让异常一路冒泡成 500。
3. 调用外部服务（LLM、支付、存储等）统一捕获网络错误和上游错误码，包装成 `LLMProviderError` 或专用异常，并在 `details` 中附带关键上下文（如第三方接口名、主要参数概要、trace id 等）。
4. 在路由层只做轻量包装和日志记录，把响应格式化交给全局异常处理器，保持 API 错误响应结构一致。
5. 对长耗时任务增加合理的轮询和超时策略，并在日志中输出阶段性状态，便于定位是「还在排队」「渲染中」还是「已失败」。

按以上约定接入，新功能可以自然融入统一的错误处理与日志框架，后期排查问题也会更加稳定和高效。
