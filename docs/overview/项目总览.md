# 项目总览（ReAngle）

本文档总览 ReAngle 的价值主张、用户流程、功能矩阵、系统协作方式与现状；各节提供指向详细文档的链接。

---

## 1. Value Proposition（项目是什么、解决什么问题）

ReAngle 的核心问题意识是：现实世界发生的是 **Events（事件）**，而人们接收到的往往是被不同立场、语气与叙事方式包装后的 **Narratives（叙事）**。同一事件可以被英雄化、道德化、丑闻化、制度化、意识形态化、娱乐化。面对信息洪流，多数人缺少把事实与观点拆开、再基于事实重建叙事的能力。

因此 ReAngle 不仅是“改写工具”，而是一个结构化的叙事工作流：在重构叙事（ReAngle）之前，先进行拆解（DeAngle）——把**事件事实**与**叙事观点**分离，并对事实进行验证，再在此基础上按用户指定的新视角生成新的文章表达。

## 2. 用户使用流程（User Journey）

用户进入主应用后，在左侧将不同来源的素材（纯文本、URL、PDF、DOCX 等）逐条加入队列；模型与改写风格在设置页配置。提交后先经 Inputs 步骤统一抽取与聚合，可选执行 DeAngle 再做 ReAngle，得到摘要与改写正文。右侧结果区域以多标签呈现：Summary、Rewritten、Compare、Avatar（数字人播报）。用户可在这些视图中阅读、对比、下载（改写正文 .txt、数字人视频），以及语音朗读或数字人视频生成。

## 3. 功能与能力矩阵（Feature Overview）

本节按 **Input → Processing → Output → Infrastructure** 四层梳理能力；细节实现与路由见对应文档。

### 3.1 Input（输入层）

**多源输入与队列处理 / Multiple input sources & queue**  
支持纯文本、URL、DOCX、PDF 等多种来源，可将多条输入加入队列，一次性完成整篇改写与摘要，适合将分散素材合并成统一上下文。

**预设提示词与场景模版 / Preset prompts & templates**  
内置“幽默化、学术化、新闻/博客风格”等预设提示词，同时允许用户自由编辑提示内容，以快速切换叙事风格。

**System Prompt 模型行为约束**  
通过系统级提示词约束模型输出行为（如结构化输出、遵循指令、减少跑题），提升改写稳定性与一致性。

**模型与风格配置 / Model & style configuration**  
界面可选择 GPT‑5、Gemini、Qwen 等模型，用户可在质量、速度、成本之间做权衡。

**多语言 UI 偏好 / Multilingual UI preference**  
界面支持中/英/西语切换，并记住用户偏好，提升跨语种用户体验。

对应文档：
- `docs/features/多源输入队列处理.md`
- `docs/features/多语言功能实现.md`

### 3.2 Processing（处理层）

**DeAngle / 事件与视角拆解**  
对 Inputs 产出的原文做 Event/Angle 拆分与事实核查，结果供 ReAngle 使用；接口 `POST /api/v2/deangle/`，结果写入 Session。

**改写与摘要 / ReAngle**  
用户选择事实与视角、填写 prompt 后，后端按用户设置中的模型生成 summary 与 rewritten_content；接口 `POST /api/v2/reangle/`。

**多模型路由 / Multi‑LLM routing**  
ReAngle/DeAngle 所用模型与系统 Prompt 在用户设置中配置，后端按设置将请求路由至对应 LLM，返回统一结构。

**异常处理 / Error handling**  
对抽取失败、参数错误、上游模型调用失败等统一封装，返回标准错误结构，并配合 request_id 与结构化日志。

### 3.3 Output（输出层）

**三视图结果展示 / Three‑view result experience**  
摘要视图：展示自动生成的摘要，并支持“一键朗读”；  
改写全文视图：展示整篇重写后文章，支持下载；  
对照视图：原文与改写稿并排，差异高亮，方便人工审阅与微调。

**文本转语音 / Text‑to‑speech (TTS)**  
将摘要生成音频并在前端播放，支持“先看摘要/先听摘要”的轻量消费方式。

**数字人播报 / Digital avatar narration**  
支持从改写结果生成可编辑口播稿，再调用 HeyGen **Video Generation API** 生成数字人视频并提供预览/下载；主播、声音、分辨率与背景通过环境变量配置，口播稿作为唯一脚本传入。前端提供 Avatar 标签入口，并按口播稿长度（如 800 字上限）控制成本与时长。

对应文档：

### 3.4 Infrastructure（基础设施层）

**技术栈与前后端分离 / Tech stack & separation**  
前端：React + Vite + TypeScript；后端：FastAPI（Python）+ 多 Provider Client。结构清晰，便于独立部署与迭代。

**身份与计费基础设施 / Auth & billing infrastructure**  
Supabase 负责身份认证与用户用量数据，Stripe 负责订阅、结算、客户门户与 Webhook；订阅状态通过 State‑Based Sync 同步到数据库，并驱动配额与权限。

**配置与环境管理 / Configuration & environments**  
集中管理环境变量（LLM/TTS/Avatar/Stripe/Supabase 等），支持本地与生产环境切换。

**日志、监控与请求追踪 / Logging, monitoring & tracing**  
请求级 request_id + 结构化日志（控制台 + 文件），让故障定位与链路追踪更直接。

## 4. 技术协作方式（模块如何调用与协同）

ReAngle 的协作方式可以理解为“前端编排体验、后端编排能力、BaaS/Stripe 承担状态与计费、外部 AI 提供生成能力”：

1) 前端负责把用户输入组织成队列与配置（模型、提示词、风格），统一提交给后端。  
2) 后端首先完成鉴权与配额控制（Supabase 校验 JWT + 原子递增用量），确保请求合法且未超限。  
3) 后端对多源内容做抽取与清洗，得到标准输入文本，再按 `llm_type` 路由到不同大模型生成摘要与改写稿，返回统一结果结构。  
4) 在结果阶段，前端用 Summary/Rewritten/Compare/Avatar 多视图承载后续交互：TTS、口播稿、HeyGen 视频生成等均作为“在同一结果上下文里的扩展动作”。  
5) 订阅计费由 Stripe 驱动，后端通过 Webhook + 权威查询同步订阅状态到 Supabase 数据库，并影响后端的 `check_usage_limit` 判定。

## 5. 总结与后续方向（Project Status & Next Steps）

当前版本已具备从“多源输入 → 多模型改写 → 多视图结果 → TTS/Avatar 扩展”的完整闭环，并配套账号、配额与订阅计费基础设施，能够支撑 MVP 级对外使用与快速迭代。

下一阶段的重要方向是把 **DeAngle** 作为一等公民能力引入主流程：先把输入拆解为 Event/Angle 并对事件做事实核查，再让 ReAngle 基于 Verified Events 与用户设定的新角度生成内容，使系统从“改写工具”升级为“叙事重构工作台”。  
（DeAngle 的详细定义与流程说明见：本目录下《DeAngle功能定义.pdf》与 `docs/overview/New Value Proposition.md`。）
